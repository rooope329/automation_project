# 「SIerの見積もりが高すぎる」と激怒のIT部長、IT部門の問題が見えているか | 日経クロステック（xTECH）

　最近、こんな話を聞いた。SIerの営業担当者がある大企業からシステムの新規開発の案件で見積もりを求められたときのエピソードだ。見積もりを提出してからしばらくして、その企業のIT部員から「IT部長が激怒している」との連絡を受けたというのだ。

　IT部長が怒った訳は、見積もりがあまりに高額だったからだ。どうやら同業他社のIT部長に話を聞いて、いかに自分たちが「ふっかけられているか」を悟ったようだ。この企業のIT部門はシステムの内製力を持たない。だから「人の足元を見やがって」という怒りにつながったのかもしれないという。

　しかし、SIerの営業担当者は「適正な」見積額を提示していたのだ。内製力を持たないそのIT部門は、システム開発だけでなく保守運用もSIerなどITベンダーに任せている。今回のシステム開発についても丸投げの意向で、提示された要件定義書も曖昧な箇所が多く、SIerの技術者による再度のヒアリングが不可欠なものだった。

　そんなわけなので、見積額が高すぎても不当なものではない。そもそも要件定義が曖昧であれば、システム開発に必要な工数を導き出せず、当初の料金は高額に設定しなければならない。SIerの技術者による再度の要件定義の工数分も、料金に上乗せする必要がある。

　見積額が高くなる理由はこれだけではない。一般論だが、SIerはこの手の企業を「危ない客」と見なして、そのリスクヘッジ分を料金に上乗せしたり、あえて高すぎる見積額を提示することで受注してしまうのを避けたりするのだ。

## 2ページ目

#### 以前なら「大炎上案件」だった

　なぜSIerがこうした企業を危ない客と見なすのかというと、システム開発プロジェクトでの手戻りや頓挫などの大きなリスクを警戒するからだ。要件定義書はどんなシステムを作ってほしいのかを文書の形で明確に示すものだ。本来ならその作成は客であるIT部門の仕事だ。

　もちろん、IT部長を激怒させたSIerは、まともな要件定義書を作成できないだけで、この企業を危ない客と見なしたわけではないだろう。同社のIT部門は技術者と言える人がいない素人集団であり、事業部門の要求を抑えられず要件が際限なく膨らむ恐れありと判断したようだ。

　結局、この商談はまとまらなかったと聞く。当然の結果と言える。そう言えば1990年代や2000年代には、SIerがこうした危ない客のシステム開発案件を請け負って、プロジェクトが破綻するといったケースが相次いだ。危ない客かどうかを見抜けなかったわけではない。危ない客と分かっていて取りにいったのだ。

　当時は、後に「失われた30年」として語られることになる日本経済の長期低迷期の前半に当たる。SIerとしても案件不足に苦しみ、技術者を「無稼働」にするよりましとして安値受注や危ない客の案件の受注に走り、プロジェクトが大炎上というのはよくあるケースだった。

　だが、今は違う。「空前の技術者不足」と言われており、SIerとしては無理をしてまで、危ない客の案件を引き受ける必要性は全くない。今回紹介した企業だけでなく、少しでも思い当たる節（ふし）があるのなら、IT部門の組織再建や技術力の強化に向けて動き出すべきだろう。

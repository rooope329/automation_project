# LLMにどこまで委ねるか、AIエージェントの「非決定論」問題 | 日経クロステック（xTECH）

　AI（人工知能）が人間に代わって業務を遂行する「AIエージェント」について、相対する意見を耳にするようになった。焦点となっているのは、業務の進め方に関する判断をLLM（大規模言語モデル）に委ねるか否かという問題だ。

#### トレンドは「LLMに判断を委ねる」

　米Anthropic（アンソロピック）の「Claude Code」や「Claude Cowork」に代表される最近人気のAIエージェントは、業務の進め方をLLMが考えてくれる。ユーザーは業務のゴール（目標）をプロンプト（指示文）として与えるだけでよい。ゴールを実現するにはどのツールをどのような順番で使用すべきかは、LLMが論理的推論（Reasoning）の能力を駆使して考えてくれる。

　今から1年前、2025年前半に普及が始まった当初のAIエージェントは、そうではなかった。AIエージェントがどのように業務を処理するか、そのワークフローはユーザーが開発する必要があった。AIエージェントのワークフローを開発するツールとして「Dify」や「Copilot Studio」などが人気となった。

#### 「汎用エージェント」になったClaude Code

　転換点となったのは、Claude Codeの台頭だ。

　Claude Codeはその名が示すとおり、ソフトウエア開発（コーディング）用の「コーディングエージェント」としてリリースされた。しかしある時からClaude Codeの熱心なユーザーが、Claude Codeを「あらゆる業務を自動化できる汎用エージェント」として使い始めたのだ。

　「例えば、財務リポートの作成を考えてみてください。モデルはAPI（アプリケーション・プログラミング・インターフェース）を呼び出してデータを取得・調査し、そのデータをファイルに整理し、Pythonで分析し、得られた洞察をファイルにまとめて記録します。これらすべてがコードを通じて行われます」

　アンソロピックのエンジニアであるMahesh Murag（マヘシュ・ムラグ）氏は、2025年11月に米ニューヨークで開催された「AI Engineer Code Summit」の講演で、コーディングエージェントによる業務自動化のイメージをこう説明している。

　AIエージェントによる業務の自動化は、最終的には業務アプリケーションなどを操作するコードに落とし込まれる。それ故に、Claude Codeなどコードをうまく生成できるコーディングエージェントさえあれば、あらゆる業務を自動化できるという理屈だ。

#### 「エージェントは開発するな」と主張するアンソロピック

　当初はClaude Codeに業務を依頼すると、Claude Codeはその業務の遂行に必要なコードを、その都度開発して実行していた。2025年10月からは、ある業務を自動化するコードは、その業務に関する専門知識と組み合わせて「Agent Skills」としてパッケージ化し、再利用できるようになった。

　2026年1月12日のClaude Coworkのリリースに続き、アンソロピックが1月末にマーケティング業務や会計業務に対応したAgent Skillsをリリースしたところ、SaaS（ソフトウエア・アズ・ア・サービス）企業の株価が軒並み下落したのは記憶に新しい。Claude Coworkに加えClaude CodeとAgent Skillsがあれば、あらゆる専門業務が自動化できると株式市場は判断したわけだ。

　「エージェントは開発するな、代わりにスキルを開発せよ（Don't Build Agents, Build Skills Instead）」。前述のムラグ氏の講演には、このような刺激的なタイトルが付けられていた。

　特定の業務に対応したAIエージェントを個別に開発するのではなく、汎用エージェントであるClaude Codeを使え、Agent Skillsを開発せよ――。講演タイトルには、そのようなメッセージが込められていた。

　Agent Skillsの仕組みは現在、米OpenAI（オープンAI）の「Codex」やオープンソースの「OpenClaw」、米Google（グーグル）の「Antigravity」など他のエージェントにも広がっている。

#### 「LLMに判断を委ねるな」との主張も根強い

　一方で、LLMが業務の進め方を判断する方式のAIエージェントについては、ミッションクリティカルな業務で使うべきではないとの批判も存在する。

　筆者が2026年5月20日に東京でインタビューした米Gartner（ガートナー）のアナリスト、Erick Brethenoux（エリック・ブレテヌー）氏は「私のクライアント、特に非常に大規模なクライアントの多くは、エージェントの内部でLLMを全く使っていない」とまで主張した。

## 2ページ目

　ブレテヌー氏がこう主張するのは、ミッションクリティカルな業務を担うエージェントは、決定論的（deterministic）な出力を行う必要があると考えているためだ。決定論的とは、ある入力に対して必ず同じ（1種類の）出力が行われる、という意味である。

　LLMは確率的に動作するため、ある入力に対する出力が異なる可能性がある。同じプロンプトを入力しても、得られる結果が異なることがあるわけだ。確率的に変動する「非決定論的（non-deterministic）」な出力は、ミッションクリティカルな業務にはふさわしくないとブレテヌー氏は指摘する。

　「あるエネルギー分野のクライアントは、『LLMのエラー率が1％だった場合、当社の環境では5000件の誤ったトランザクションが発生することになり、毎年5人が死亡する恐れがある。そのようなエラー率は許容できない』と語っている。製品のレコメンデーションを行うアプリケーションなら5％のエラー率でも許容できるだろうが、人々の命を預かるエネルギー分野やインフラ分野ではそうはならない」。ブレテヌー氏はそう力説する。

　ブレテヌー氏はAIエージェントにはLLMではなく「ナレッジグラフや従来型の機械学習モデル、ルールベース、最適化モデルなど従来型のAIを使うべきだ。従来型のAIは信頼できるAIであり、これらをエージェントに組み込めば、エージェントが自律的に行う作業を信頼できる」と主張する。

#### AIエージェントの高い自律性にはリスクもある

　実はブレテヌー氏と同様の主張は、米英豪などの情報機関も発している。

　米英豪などの機密情報共有グループ「ファイブアイズ」に参加する5カ国の情報機関が2026年5月1日（日本時間）に公開した「Careful adoption of agentic AI services（エージェント型AIサービスの慎重な導入）」と題するリポートだ。

[「AIエージェントは慎重に導入せよ」、米英豪などの情報機関が警告](/atcl/nxt/column/18/03079/050700036/?i_cid=nbpnxt_sied_blogcard)

　AI（人工知能）が自律的に判断して仕事を進めるAIエージェントにはセキュリティーリスクが多数あるため、慎重に導入すべきだ――。米英豪などの機密情報共有グループ「ファイブアイズ」に参加する5カ国の情報機関が2026年5月1日、このような内容のリポートを公開した。

　同リポートは各国の政府機関に対して「エージェント型AIは、反復的で明確に定義された低リスクのタスクを自動化できる」「エージェント型AIよりもリスクが低い可能性があるソリューションをなるべく検討すべきだ」と主張する。

　LLMの論理的推論能力をベースにしたAIエージェントの自律性が、政府機関にとってセキュリティーリスクになり得るとの危機感があるためだ。

　判断をLLMに委ねるタイプのAIエージェントは、汎用的な利用が可能であり、ユーザーにとって便利であるのは間違いない。その一方である程度のリスクが存在するのも確かだ。非決定論的なLLMの挙動をどこまで受け入れられるのか。ユーザー企業が正しく判断する必要がある。
